Product Definition

这页到底在做什么

它不是一个“医学搜索框”，而是一层可追溯的循证服务，把问题拆成证据检索、指南对照、临床试验匹配和药物安全复核四件事。

Target Users目标用户

医生 Copilot、医学事务团队、临床科研助理、专科知识平台、研究生训练项目。

临床科研医学事务内容平台
Primary Outputs主要输出

结构化证据摘要、带来源引用的问答、正在招募的临床试验列表、OpenFDA 风险信号提示。

引用证据等级试验匹配安全提示
Success Criteria成功标准

所有回答都有来源、可回链、可复查，并明确区分“指南推荐”“研究发现”和“模型总结”。

可追溯可复核可更新有边界

Workflow

从提问到交付的五步工作流

页面适合直接展示给产品、医院或合作方，说明系统怎么把自然语言问题转成证据化输出。

STEP 1question normalization

把用户问题规范成疾病、疗法、终点、患者特征和时间窗，必要时转成英文医学术语。

STEP 2multi-source retrieval

并行检索 PubMed、ClinicalTrials.gov、OpenFDA、OpenTargets 和权威指南源，避免单源回答。

STEP 3evidence synthesis

把综述、RCT、试验招募、指南语句和安全信号合并为一份结构化证据摘要。

STEP 4grade and compare

标记证据类型、来源日期和推荐等级，区分“标准治疗”“探索性研究”和“风险提醒”。

STEP 5human review

最终输出进入医生或研究员复核，不直接代替临床决策，保留审计日志和源链接。

Delivery Plan

建议的三阶段落地

同一个场景可以先做成内部工作台，再做成 API 服务，最后再嵌入更多业务系统。

Phase 1内部证据工作台

先给研究助理或医学事务团队使用，验证问题类型、查询频次和输出格式。

Phase 2统一 Evidence API

将证据检索、指南对照和试验匹配服务化，供医生 Copilot、内容平台或问答系统调用。

Phase 3嵌入式临床知识层

与 AI-native HIS、AI-native Up2date 联动，形成“问题 - 证据 - 决策支持 - 复核”的闭环。

Governance

必须明确写出的边界

医学场景里，页面完整不只是内容多，更重要的是把不能做什么也写清楚。

Not Diagnosis不直接给个体诊疗结论

输出是证据整理和来源引用，不代替床旁治疗决策。

Freshness标记来源时间

所有指南、文章和试验都应显示日期或检索时间，避免“最新”被误解。

Audit保留审计链

记录问题、使用的工具、返回来源和人工复核结果，方便后续质量追踪。
