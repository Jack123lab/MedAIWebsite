虚拟医院

虚拟医院不是一个“聊天病人”，而是一个由数字孪生患者、病区流程、MDT 协作和教学复盘组成的模拟环境，适合住培教学、方案演练和新流程验证。

Core Modules

虚拟医院的四个核心模块

这页适合直接讲产品结构，说明用户进入系统后到底会看到哪些空间和工具。

Outpatient模拟门诊

医生或学员围绕主诉、病史、检查选择和解释沟通展开问诊，系统记录全流程。

Ward虚拟病区

用数字孪生患者承接病程变化、治疗反馈和随时间演进的检验与影像结果。

MDT多学科讨论室

让不同角色围绕同一病例提出判断、补充证据和协商方案，适合训练协作。

Workflow

一个训练案例如何在系统里跑完

把流程写清楚，页面才像完整产品，而不只是“可以模拟病例”的一句话。

STEP 1case import

从脱敏病例、教学案例或研究样本导入基础资料，生成可演练的初始患者状态。

STEP 2twin initialization

按病史、化验、影像和时间轴初始化数字孪生患者，决定哪些信息可见、哪些需通过追问获取。

STEP 3interaction

学员进行问诊、下检查、提出诊断和治疗，系统根据预设病程或模拟模型反馈结果。

STEP 4mdt / compare

教师或多角色代理进入 MDT 环节，对照不同方案、解释依据、比较路径优劣。

STEP 5debrief

系统输出复盘报告，记录错过的关键信号、提问质量、证据使用情况和沟通表现。

SCENARIO

推荐展示的典型训练案例

用一个页面说明虚拟医院“怎么被用”，通常比单纯堆技术名词更容易被理解。

住培MDT病例回放沟通训练
案例：儿童呼吸道感染复诊场景

1. 学员进入门诊接诊，完成问诊与风险识别
2. 系统按病史反馈体温、化验和影像结果
3. 学员给出初步诊断与处理建议
4. 教师切换到 MDT 视角，比较两种处置路径
5. 系统输出复盘：遗漏提问、检查冗余、沟通表述和风险提示

Delivery

适合怎样落地

虚拟医院通常先从教学和流程验证起步，再逐渐扩展到研究沙盒或更复杂的数字孪生平台。

Phase 1教学训练版

用脱敏病例和规则化反馈建立最基础的住培或研究生训练环境。

Phase 2病例回放版

加入病程时间轴、检查回看和 MDT 讨论，用于复杂案例复盘与流程复训。

Phase 3数字孪生版

引入更多真实世界数据与模型，形成面向研究与模拟验证的虚拟医院平台。

Boundary只用于训练与模拟

页面必须清楚标注不用于真实患者即时诊疗，不输出真实医嘱。

Data只用脱敏或模拟数据

病例、病历和检查结果都需脱敏，避免真实身份信息进入演练环境。

Review复盘优先于评分

系统最重要的价值是帮助训练者看清思路和盲点，而不是只给一个分数。
